# WikiStreams 서비스 수준 협약 (SLA)

> 작성일: 2026-03-06
> 상태: **초기 협약** (SLO 관측 기간 완료 후 수치 확정 예정)
> 선행 산출물: NFR.md, SLI.md, SLO.md
> 적용 범위: WikiStreams 홈랩 데이터 파이프라인 운영자(개인)

---

## 1. SLA 개요

### 1.1 목적

SLO.md에 정의된 서비스 수준 목표를 기반으로, 서비스 운영자(본인)가 자기 자신에게 약속하는 **책임 범위, 위반 처리 절차, 보고 주기**를 명시한다.

이 SLA는 외부 고객과의 계약이 아닌 **홈랩 운영 규율(self-imposed discipline)**이며, 운영 품질 추적과 개선의 기준점으로 활용한다.

### 1.2 당사자

| 역할 | 내용 |
|------|------|
| 서비스 제공자 | WikiStreams 운영자 (개인) |
| 서비스 사용자 | 운영자 본인, Grafana 대시보드 열람자 |

### 1.3 서비스 범위

| 서비스 | 범위 |
|--------|------|
| 실시간 이벤트 수집 | Wikimedia SSE → Kafka → ClickHouse 파이프라인 |
| 대시보드 | Grafana Analytics, SLO, Producer, Resources 대시보드 |
| 일일 리포트 | Discord Embed 리포트 (매일 09:00 KST) |
| 알림 | Discord SLO 위반 알림 |

## 2. 서비스 가용성 약속

### 2.1 가용성 목표

| 서비스 | SLO | 월간 허용 다운타임 |
|--------|-----|-------------------|
| ClickHouse (쿼리 가능) | 99% | 7.2시간 / 30일 |
| 일일 Discord 리포트 | 90% (월 27회 이상) | 3회 누락 허용 |

### 2.2 외부 의존성 제외

아래 외부 서비스 장애로 인한 SLO 위반은 가용성 계산에서 제외한다:

- Wikimedia SSE API 장애 (이벤트 수집 중단)
- Wikidata REST API 장애 (레이블 보강 불가)
- Anthropic Claude API 장애 (리포트 생성 불가)
- Discord Webhook 장애 (리포트/알림 발송 불가)
- Google News RSS 장애 (뉴스 섹션 누락)
- S3 Compatible Storage 장애 (백업 불가)

외부 장애 여부는 해당 서비스의 공식 상태 페이지 또는 로그 패턴으로 판단한다.

## 4. SLO 위반 처리 절차

### 4.1 위반 감지

| 방법 | 내용 |
|------|------|
| **자동 감지** | Grafana Alert Rule → Discord 알림 (SLI-D1, SLI-A3, SLI-R1, SLI-P7, SLI-CAP1, SLI-CAP2) |
| **수동 확인** | 월 1회 SLO 대시보드 검토 (에러 버짓 잔량, 30일 달성률) |

### 4.2 대응 우선순위

| 심각도 | 조건 | 대응 목표 시간 |
|--------|------|----------------|
| **critical** | ClickHouse 메모리 > 80% / 데이터 신선도 lag > 5분 | 30분 이내 확인 및 조치 시작 |
| **warning** | SLO 위반 감지 (D1, R1, P7, CAP2, A3) | 당일 내 원인 파악 |
| **info** | 에러 버짓 소진률 75% 초과 | 주간 리뷰 시 검토 |

### 4.3 위반 처리 흐름

```
Discord 알림 수신
  → Grafana SLO 대시보드 확인 (어떤 SLI인지 파악)
  → 원인 분석 (Loki 로그, ClickHouse query_log, resource-monitor)
  → 조치 (서비스 재시작 / 설정 변경 / 운영자 개입)
  → DEV_LOG.md에 장애 내용 기록
  → 월간 SLO 리뷰 시 재발 방지책 검토
```

### 4.4 장애 기록 기준

아래 조건 중 하나라도 해당하면 `docs/DEV_LOG.md`에 장애 항목 추가:
- SLO 위반 지속 시간 ≥ 30분
- 일일 리포트 누락
- ClickHouse 데이터 손실 발생
- 에러 버짓 소진률 100% 도달

## 5. 데이터 보존 및 복구 약속

### 5.1 데이터 보존

| 데이터 | 보존 기간 | 저장 위치 |
|--------|----------|-----------|
| ClickHouse 이벤트 | 90일 (TTL) | Docker volume `clickhouse_data` |
| Loki 로그 | 30일 (retention) | Docker volume `loki_data` |
| Wikidata SQLite 캐시 | 30일 TTL (정상 엔티티) | Docker volume `producer_cache` |
| Discord 리포트 JSON | 무기한 | `./reports/` 디렉토리 |

### 5.2 복구 목표 (S3 백업 구현 후 적용)

| 지표 | 목표 | 비고 |
|------|------|------|
| RPO (Recovery Point Objective) | ≤ 1시간 | 시간당 증분 백업 주기 기준 |
| RTO (Recovery Time Objective) | ≤ 30분 | S3 복원 + 컨테이너 재시작 포함 |

**현재 상태**: S3 백업 미구현. 로컬 Docker volume만 존재. 호스트 장애 시 데이터 손실 위험.
→ S3 Datalake 도입(TODO.md 참조) 완료 전까지 RPO/RTO SLA 미적용.
