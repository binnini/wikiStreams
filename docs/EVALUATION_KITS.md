# **평가 질문 스타터킷**

## Data Source System Evaluation

---

- 데이터 원천의 본질적인 특징은 무엇인가? **데이터 원천은 애플리케이션인가? loT 장치의 스웜인가?**
    - 설명
        
        원천이 '웹/앱 애플리케이션의 백엔드 DB'라면 정형화된 데이터일 확률이 높고 일괄 처리(Batch)가 유리할 수 있다. 반면, '수만 개의 IoT 센서'라면 데이터가 끊임없이 들어오는 스트리밍(Streaming) 환경이며, 네트워크 단절 등에 대비한 아키텍처(예: Kafka, MQTT 도입)가 필요하다.
        
- **원천 시스템에서 데이터는 어떻게 유지되는가?** 데이터는 장기간 보존되는가? 아니면 일시적이고 빠르게 삭제되는가?
    - 설명
        
        로그 파일처럼 데이터가 1시간 뒤에 삭제되거나 덮어씌워진다면, 파이프라인은 그 시간 안에 반드시 데이터를 가져와야 합니다(실패 시 데이터 유실 발생). 반면 DB에 수년간 보존되는 데이터라면, 파이프라인이 멈춰도 나중에 다시 소급 적용(Backfill)하기가 매우 수월합니다.
        
- **데이터는 어느 정도의 속도로 생성되는가?** 초당 몇 개의 이벤트가 발생할까? 시간당 몇 기가바이트 인가?
    - 설명
        
        파이프라인의 '인프라 용량'을 결정합니다. 시간당 메가바이트 수준이면 간단한 파이썬 스크립트와 크론(Cron)으로 충분하지만, 초당 수십만 건의 이벤트가 발생한다면 Spark나 Flink 같은 분산 처리 프레임워크와 오토스케일링이 필수적입니다.
        
- **데이터 엔지니어는 출력 데이터에서 어느 정도의 일관성을 기대할 수 있는가?** 출력 데이터에 대해 데이터 품질 검사를 실행할 때, 예상치 못한 출력값（예를 들면 n니II 값）이나 잘못된 데이터 포맷과 같은 데이터 불일치 사례는 얼마나 자주 발생할까?
    - 설명
        
        "Garbage In, Garbage Out"을 방지하기 위함입니다. 일관성이 낮다면 파이프라인 중간에 강력한 데이터 정제(Cleansing) 로직과 검증 단계가 필요하며, 잘못된 데이터를 따로 격리하는 **데드 레터 큐(Dead Letter Queue) 구축**을 고려해야 합니다.
        
- **에러는 얼마나 자주 발생하는가?**
    - 설명
        
        API 타임아웃이나 연결 끊김이 잦다면, 데이터 추출 코드에 **재시도 로직(Exponential Backoff 등)**을 견고하게 짜야 하며, 에러 발생 시 알림(Alerting) 임계치를 어떻게 설정할지 결정하는 기준이 됩니다.
        
- **데이터에 중복이 포함되지는 않는가?**
    - 설명
        
        데이터 파이프라인은 **'멱등성(Idempotency)**'을 가져야 합니다. 원천에서 중복 데이터를 보낼 수 있다면, 엔지니어는 고유 ID나 타임스탬프를 활용해 중복을 제거(Deduplication)하는 로직을 반드시 구현해야 합니다.
        
- 일부 데이터값이 동시에 생성되는 다른 메시지보다 훨씬 늦게 도착할수 있는가?
    - 설명
        
        **지연 도착(Late-arriving)** 또는 **순서가 뒤섞인(Out-of-order)** 데이터의 존재 여부를 묻습니다. 오프라인 상태의 모바일 기기가 온라인이 될 때 과거 데이터를 한 번에 보내는 경우가 대표적입니다. **스트리밍 처리에서는 '워터마크(Watermark)' 개념**을 도입해 지연 데이터를 기다릴지 버릴지 결정해야 하며, **배치 처리에서는 과거 파티션을 다시 덮어쓰는 로직**이 필요합니다.
        
- **수집된 데이터의 스키마는 무엇인가?** 데이터 엔지니어가 데이터를 완전히 파악하려면 여러 테이블 또는 여러 시스템에 걸쳐 조인을 수행해야 하는가?
    - 설명
        
        데이터 추출(Extraction)의 복잡도를 결정합니다. 조인이 많이 필요하다면 원천 시스템에 부하를 줄 수 있으므로, 원형 그대로(Raw) 데이터 웨어하우스로 먼저 가져온 뒤 그 안에서 조인하는 **ELT(Extract-Load-Transform) 방식**을 채택하는 것이 유리합니다.
        
- 스키마가 변경되면（예를 들어 새로운 열이 추가되었을 때) 어떻게 대처하고 다운스트림 이해관계자에게 전달할 수 있는가?
    - 설명
        
        파이프라인이 깨지는 가장 흔한 원인입니다. **스키마 레지스트리(Schema Registry)**를 사용해 유연하게 대처하거나, 백엔드 개발자(데이터 생산자)가 스키마 변경 전 데이터 엔지니어 팀에 미리 알리도록 **커뮤니케이션 프로세스와 계약(Data Contract)**을 수립해야 합니다.
        
- **원천 시스템에서 데이터를 얼마나 자주 가져와야 하는가?**
    - 설명
        
        비즈니스 요구사항(예: "대시보드는 실시간이어야 한다" vs "하루 전 데이터면 충분하다")과 인프라 비용 사이의 타협점을 찾는 과정입니다. **이에 따라 실시간 스트리밍, 마이크로 배치, 일별 배치 등 수집 방식을 결정**합니다.
        
- **Stateful system의 경우(고객 계정 정보를 추적하는 데이터베이스 등), 데이터는 정기적인 스냅숏으로 제공되는가? 아니면 change data capture（CDC）로부터의 갱신 이벤트로 제공되는가?** 변경은 어떻게 수행되며, 원천 데이터베이스에서 이러한 변경을 어떻게 추적하는가?
    - 설명
        - **스냅숏:** 매번 테이블 전체를 복사합니다. 구현은 쉽지만 데이터가 커질수록 엄청난 시간과 부하가 발생합니다.
        - **CDC:** INSERT, UPDATE, DELETE 등 변경된 내역(로그)만 실시간으로 가져옵니다(예: Debezium 활용). 효율적이고 실시간 처리가 가능하지만, 구축 난이도가 높습니다.
- 다운스트림 사용을 위한 데이터를 전송하는 데이터 제공업체는 누구（무엇）인가?
    - 설명
        
        사내 타 부서인지, 외부 서드파티 API 업체인지에 따라 권한(Authentication) 획득 방식, API 호출 제한(Rate Limit), 장애 발생 시 연락망 및 **SLA(서비스 수준 협약) 기준**이 완전히 달라집니다.
        
- **데이터 원천에서의 데이터 조회가 성능에 영향을 미치는가?**
    - 설명
        
        **우리가 데이터를 추출하는 행위가 원천 시스템(보통 서비스 운영 DB)에 부하를 주는지 묻는 매우 중요한 질문입니다.** 분석용 쿼리 때문에 실제 서비스가 다운되면 안 됩니다. **부하가 크다면 운영 DB(Primary)에 직접 접근하지 않고, 읽기 전용 복제본(Read Replica)에서 데이터를 가져오거나 시스템 사용량이 적은 새벽 시간에만 배치를 돌려야 합니다.**
        
- **원천 시스템에 업스트림 데이터 의존 관계가 있는가**? 이러한 업스트림 시스템의 특징은 무엇인가?
    - 설명
        
        우리가 바라보는 데이터 원천이, 사실은 또 다른 시스템에서 데이터를 받아오는 곳인지(데이터 리니지) 묻습니다. 업스트림에서 파이프라인이 멈추거나 지연되면, 우리가 가져오는 데이터는 **'오래된 데이터(Stale data)'**가 됩니다. 데이터의 흐름과 의존성을 파악해야 **전체적인 데이터 파이프라인 스케줄링(예: Airflow의 DAG 구성)**을 올바르게 할 수 있습니다.
        
- 늦거나 누락된 데이터 확인용으로 데이터 품질 검사가 실시되고 있는가?
    - 설명
        
        장애를 유저나 비즈니스 팀보다 엔지니어가 먼저 알아채기 위한 방어선입니다. 평소 하루 10GB씩 들어오던 데이터가 갑자기 1GB만 들어온다면(Anomaly), **파이프라인 자체는 성공하더라도 알림을 발생시키는 모니터링 체계가 필요**합니다.
        

## Storage System Evaluation

---

- **이 스토리지 설루션은 아키텍처에서 요구하는 쓰기 및 읽기 속도와 잘 맞는가?**
    - 설명
        
        스토리지의 **처리량(Throughput)**과 **지연 시간(Latency)**이 프로젝트의 요구사항을 충족하는지 묻는 질문입니다. 초당 수십만 건의 로그를 기록해야 한다면 쓰기 속도가 압도적으로 빠른 스토리지가 필요합니다. 반면, 분석가가 복잡한 쿼리를 빠르게 조회해야 한다면 읽기 속도(특히 집계 성능)가 뛰어난 스토리지를 선택해야 합니다. **워크로드의 성격(OLTP vs OLAP)에 따라 적절한 시스템을 매칭하는 첫 단계입니다.**
        
- **스토리지가 다운스트림 프로세스의 병목 현상을 초래하지는 않는가?**
    - 설명
        
        **스토리지가 전체 데이터 파이프라인의 속도를 갉아먹는 '가장 느린 구간'이 되지 않는지 확인**합니다. 데이터를 가져와서(Ingest) 처리(Process)하는 시스템이 아무리 강력해도, 최종적으로 데이터를 저장하거나 다시 읽어갈 스토리지의 I/O 성능이 떨어지면 다음 단계(머신러닝 모델 학습, 대시보드 렌더링 등)가 모두 지연됩니다.
        
- **이 스토리지 기술이 작동하는 방식을 인지하고 있는가?** 스토리지 시스템을 최적으로 활용하는가? 아니면 부자연스러운 행동을 하는가?
    - 예를 들어 객체 스토리지 시스템에 높은 비율의 임의 접근random access 갱신을 적용하고 있지는 않은가? (이것은 성능 오버헤드가 큰 안티패턴anti-pattern이다)
    - **AWS S3나 Google Cloud Storage 같은 '객체 스토리지'는 파일 전체를 덮어쓰거나 새로 추가하는 데는 탁월하지만, 파일 안의 특정 데이터 1줄만 수정(Update)하는 작업에는 최악의 성능을 보입니다**
    - 설명
        
        선택한 스토리지의 근본적인 아키텍처를 이해하고, 그에 맞는 올바른 방식으로 사용하고 있는지 묻는 질문입니다. 기술의 장단점을 명확히 알고 목적에 맞게 써야 비용 폭탄이나 성능 저하를 막을 수 있습니다. 
        
- **이 스토리지 시스템은 향후 예상되는 확장을 처리할 수 있는가?**
    - 사용 가능한 총 스토리지, 읽기 작업 속도, 쓰기 볼륨 등 스토리지 시스템의 모든 용량 제한을 고려해야 한다.
    - 설명
        
        데이터 볼륨, 읽기/쓰기 요청 횟수가 급증할 때 스토리지가 이를 버틸 수 있는지(**Scalability**) 평가합니다. 비즈니스가 성장해 내년에 데이터가 10배 늘어났을 때, 스토리지를 통째로 갈아엎어야 한다면 실패한 설계입니다. 서버를 추가해 무한히 용량을 늘릴 수 있는 **수평적 확장(Scale-out)**이 지원되는지, **IOPS(초당 입출력 횟수) 제한**은 없는지 미리 계산해야 합니다.
        
- 다운스트림 사용자와 프로세스가 필요한 **서비스 수준 협약service-level agreement （SLA）**에 따라 데이터를 취득할 수 있는가?
    - 설명
        
        데이터 소비자가 요구하는 서비스 수준 협약(SLA, 예: 99.99% 가동률, 1초 이내 응답 등)을 스토리지가 보장할 수 있는지 확인합니다. 경영진이 보는 핵심 매출 대시보드에 연결된 스토리지가 하루에 한 번씩 다운되거나 쿼리에 5분이 걸린다면 SLA 위반입니다. 요구되는 **신뢰성과 응답 속도에 맞춰 고가용성(High Availability) 아키텍처**가 구성되어야 합니다.
        
- 스키마 진화, 데이터 흐름, 데이터 계보 등에 대한 **메타데이터metadata를 캡처하고 있는가?**
    - 메타데이터는 데이터 활용성에 큰 영향을 미친다. 메타데이터는 미래에 대한 투자로. 검색 가능성과 제도적 지식을 획기적으로 향상시켜 미래의 프로젝트 및 아키텍처 변경을 간소화한다.
    - 설명
        
        데이터가 어디서 왔고**(Data Lineage)**, 스키마가 언제 어떻게 변했는지 기록하지 않으면 스토리지는 곧 아무도 쓸 수 없는 **'데이터 늪(Data Swamp)**'이 됩니다. 데이터 카탈로그 도구와 연동하여 데이터를 쉽게 검색하고 신뢰할 수 있도록 만드는 미래지향적 작업입니다.
        
- 순수 스토리지 설루션（객체 스토리지）인가? 아니면 복잡한 쿼리 패턴（예: 클라우드 데이터 웨어하우스)을 지원하는가?
    - 설명
        
        단순히 파일만 저장하는 곳인지(**Data Lake**), 아니면 저장과 동시에 강력한 연산/조회 기능까지 제공하는지(**Data Warehouse**)를 구분합니다. **S3 같은 객체 스토리지는 저장 비용이 저렴하지만 자체적으로 복잡한 SQL을 실행하기는 어렵습니다**(외부 컴퓨팅 엔진 필요). 반면 **Snowflake나 BigQuery 같은 클라우드 데이터 웨어하우스는 스토리지와 컴퓨팅이 결합되어 있어 즉각적이고 복잡한 분석에 유리**합니다.
        
- 스토리지 시스템이 **스키마**에 구애받지는 않는가? (객체 스토리지) 유연한 스키마 (카산드라)인가? 강제 적용된 스키마（클라우드 데이터 웨어하우스）인가?
    - 설명
        - **스키마 구애 없음 (객체 스토리지):** 아무 형태의 파일(JSON, 이미지, 비디오 등)이나 일단 저장하고 봅니다 (Schema-on-read).
        - **유연한 스키마 (NoSQL, 예: Cassandra, MongoDB):** 구조가 자주 바뀌는 비정형/반정형 데이터에 유리합니다.
        - **강제 적용 스키마 (데이터 웨어하우스):** 정해진 테이블 구조에 맞지 않는 데이터는 저장을 거부합니다 (Schema-on-write). 분석 데이터의 품질을 엄격하게 관리할 때 씁니다.
- 데이터 거버넌스data governance를 위해 마스터 데이터, 골든 레코드 데이터 품질 및 데이터 계보를 어떻게 추적하고 있는가?
    - 설명
        
        동일한 고객 정보가 A 시스템과 B 시스템에서 다르게 저장되어 있을 때, 스토리지 계층에서 어떤 데이터를 '진짜(Golden Record)'로 취급할지 정책을 세우고 접근 권한을 제어(Governance)하는 체계가 필요합니다.
        
- 법령 준수 및 데이터 주권에 어떻게 대처하고 있는가? 예를 들어 특정 지리적 위치에는 데이터를 저장 하고 다른 위치에는 저장하지 않을 수 있는가?
    - 설명
        
        GDPR(유럽 개인정보보호법) 같은 규제는 유럽 시민의 데이터를 반드시 유럽 내 서버에 저장하도록 강제합니다. 글로벌 서비스를 구축할 때, 스토리지 시스템이 리전(Region)별 데이터 격리와 민감 정보 마스킹/암호화 기능을 기본적으로 제공하는지 반드시 따져봐야 합니다.
        

## Data Ingestion System Evaluation

---

- **수집 중인 데이터의 사용 사례는 무엇인가?** 같은 데이터셋의 여러 버전을 생성하는 대신 이 데이터를 재사용할 수 있는가?
    - 설명
        
        데이터를 '**사일로(Silo, 고립된 상태)**'로 만들지 않기 위한 질문입니다. A팀과 B팀이 같은 고객 데이터를 필요로 할 때, 수집 파이프라인을 두 개 만드는 것은 낭비입니다. **한 번 수집해서 중앙(Data Lake 등)에 두고 여러 다운스트림 프로세스가 재사용(Single Source of Truth)할 수 있도록 설계**해야 합니다.
        
- 시스템이 이 데이터를 안정적으로 생성하고 수집하고 있는가? **필요할 때 해당 데이터를 사용할 수 있는가?**
    - 설명
        
        수집 파이프라인의 **내결함성(Fault Tolerance)**을 묻는 것입니다. 네트워크 장애로 데이터 수집이 잠시 끊기더라도 복구 후 누락 없이 다시 가져올 수 있는지, 파이프라인이 **멱등성을 가지도록 설계되었는지** 확인해야 합니다.
        
- **수집 후 데이터 목적지destinatton는 어디인가?**
    - 설명
        
        수집한 데이터를 **S3 같은 객체 스토리지(Data Lake)**에 바로 떨굴 것인지, **Snowflake 같은 데이터 웨어하우스**에 넣을 것인지, 아니면 **Kafka 같은 메시지 큐**로 보낼 것인지 결정합니다. 목적지에 따라 수집 도구(커넥터)의 종류와 네트워크 보안 설정이 완전히 달라집니다.
        
- **데이터에 얼마나 자주 접근해야 하는가?**
    - 설명
        
        **스토리지의 티어링(Tiering) 및 캐싱 전략**과 연관됩니다. 수집 직후 실시간 대시보드에서 1초마다 조회하는 **핫(Hot) 데이터**라면 **빠른 인메모리 DB나 캐시 계층**을 거치게 하고, 1년에 한 번 보는 **콜드(Cold) 데이터라면 저렴한 아카이브 스토리지**로 바로 넘기는 수집 경로를 짭니다.
        
- **데이터는 보통 어느 정도의 용량으로 도착하는가?**
    - 설명
        
        **네트워크 대역폭(Bandwidth)과 수집 서버의 크기를 결정**합니다. **트래픽 스파이크(갑자기 데이터가 몰림)가 예상된다면, 수집 서버 앞단에 버퍼(Buffer) 역할을 하는 큐를 두거나 오토스케일링 그룹을 설정해야 서버 다운을 막을 수 있습니다.**
        
- **데이터 형식은 무엇인가? 다운스트림 스토리지 및 변환 시스템에서 이 형식을 처리할수 있는가?**
    - 설명
        
        **원천 데이터가 JSON, CSV, 혹은 바이너리(Avro, Protobuf)인지 파악**합니다. 수집 목적지가 특정 포맷만 지원한다면 수집 단계에서 **직렬화/역직렬화(Serialization)**를 수행해야 하며, **스키마 레지스트리를 통해 포맷 변경에 대응**해야 합니다.
        
- **원천 데이터는 다운스트림에서 즉시 사용할 수 있는 양호한 상태인가?** 그렇다면 얼마나 오래 사용할 수있으며, 사용할 수 없게 되는 요인은 무엇인가?
    - 설명
        
        **ELT(수집 후 변환)를 할지, ETL(변환 후 수집)을 할지 결정하는 기준**입니다. 데이터가 너무 지저분하거나 민감 정보(주민번호 등)가 섞여 있다면, 목적지에 도달하기 전에 수집 단계에서 마스킹하거나 정제하는 가벼운 변환 작업이 선행되어야 합니다.
        
- **데이터가 스트리밍 소스에서 전송된 경우, 목적지에 도달하기 전에 데이터를 변환해야 하는가? 데이터가 스트림 자체 내에서 변환되는 형태의 변환이 적절할까?**
    - 설명
        
        수집 중인 데이터(In-flight data)를 실시간으로 조작할 필요가 있는지 묻습니다. 예를 들어 센서에서 화씨로 들어오는 온도를 섭씨로 즉각 변환해야 한다면 단순 수집 도구가 아니라 **Flink, Spark Streaming 같은 스트림 처리 엔진이 필요**해집니다.
        
    - **Streaming vs Batch**
        - 스트리밍 수집의 사용 사례로는 무엇이 있을까?
        - 스트리밍을 구현하면 구체적으로 어떤 이점을 얻을 수있을까?
        - **데이터를 실시간으로 가져올 수 있다면, 배치 방식에 비해 개선될 수 있는 데이터에 대해 어떤 조치를 취할 수 있을까?**
        - 스트리밍 우선 접근 방식은 단순 배치 방식보다 시간, 비용, 유지 보수, 다운타임 및 기회비용 측면에서 더 많은 비용을 소비할까?
            - 설명
                
                스트리밍 아키텍처는 항상 켜져 있어야 하므로 인프라 비용이 높고, 디버깅이 어려우며, 지연 데이터 처리(**Watermarking** 등) 로직이 복잡해 유지보수 난이도가 뜁니다. 실시간의 가치가 이 높은 도입/운영 비용(Trade-off)을 상쇄하는지 냉정하게 평가해야 합니다.
                
        - **인프라에 장애가 발생했을 때 스트리밍 파이프라인과 시스템이 안정적이고 다중화되어 있는가?**
            - 설명
                
                배치는 실패하면 다시 돌리면 그만이지만, **스트리밍은 노드가 죽었을 때 데이터를 유실하거나 중복 처리할 위험**이 있습니다. '정확히 한 번(**Exactly-once**) 처리'를 보장하기 위한 체크포인트 및 장애 조치(**Failover**) 설계가 필수입니다.
                
        - **사용 사례에 가장 적합한 도구는 무엇인가? 관리형 서비스(아마존 키네시스#=n Kines.s, 구글 클라우드 Pub/Sdb, 구글 클라우드 데이터플로 =* Cloud Datatow)를 사용해야 하는가? 아니면 카프카, 플링크스 파크, 펄사 등의 인스턴스를 구축해야 할까? 후자를 선택핸다면 누가 관리의 역할을 맡을 것인가? 비용과 트레이드오프는 무엇일까?**
            - 설명
                
                운영 부담을 누가 질 것인지의 문제입니다. **EC2 인스턴스에 Kafka 클러스터를 직접 올리면 세밀한 튜닝이 가능하지만, Zookeeper/KRaft 관리와 파티션 리밸런싱 등 엄청난 DevOps 리소스가 투입**됩니다. 반면 **AWS MSK, Kinesis, Google Pub/Sub 같은 관리형 서비스(Managed Service)를 쓰면 인프라 관리 부담을 돈으로 해결**할 수 있습니다. 팀의 역량과 예산에 맞춘 전략적 선택이 필요합니다.
                
        - ML 모델을 배포했을 때 온라인 예측 및 지속적인 훈련으로 얻을 수 있는 이점은 무엇일까?
            - 설명
                
                **스트리밍 데이터를 ML에 결합하면 과거 데이터로 훈련된 정적인 모델이 아니라, 현재 트렌드를 실시간으로 반영하는 동적인 시스템**을 만들 수 있습니다. 데이터 파이프라인이 단순 서빙을 넘어 **실시간 피처 스토어(Feature Store) 역할**까지 겸하게 됩니다.
                
        - 실제 운영 인스턴스에서 데이터를 가져오는가? 그렇다면 이 원천 시스템에 대한 수집 프로세스의 영향도는 얼마나 될까?
            - 설명
            운영 DB에 실시간 쿼리를 계속 날리면 서비스가 뻗어버릴 수 있습니다. 따라서 스트리밍 환경에서는 DB의 성능 저하 없이 변경 로그만 조용히 가로채는 **CDC(Change Data Capture, 예: Debezium)** 기술을 도입하는 것이 핵심입니다.
    

## Data Transformation System Evaluation

---

- **변환에 드는 비용과 투자수익률(ROI)은 얼마인가?** 관련된 비즈니스 가치는 무엇인가?
    - 설명
        
        데이터 웨어하우스(예: BigQuery, Snowflake)에서 수백 기가바이트의 데이터를 조인(Join)하고 집계(Aggregation)하는 쿼리를 매일 돌리면 막대한 클라우드 비용이 발생합니다.
        
        - 예를 들어, 반도체 공정 데이터를 분석할 때 수만 개의 센서 로그를 매초 단위로 롤업(Roll-up)하여 웨이퍼 불량률을 예측하는 변환 파이프라인을 만든다고 가정해 보겠습니다. 이 파이프라인 연산에 한 달에 100만 원이 들더라도, 불량품을 조기에 발견해 수천만 원의 원가를 절감할 수 있다면 **ROI가 확실한 훌륭한 변환**입니다.
        - 반면, 아무도 보지 않는 대시보드를 위해 매시간 무거운 해시 연산을 돌린다면 이는 엔지니어링 리소스와 인프라 비용의 낭비입니다. 데이터 엔지니어는 쿼리 최적화뿐만 아니라, ‘이 변환이 비즈니스에 필요한가’를 비판적으로 따져야 합니다.
    
- **변환은 가능한 한 단순하고 독립적인가?**
    - 설명
        
        변환 로직이 스파게티 코드처럼 복잡하게 엉켜있지 않고, 모듈화(Modularization)되어 있어 유지보수와 디버깅이 쉬운지 평가합니다.
        
        - 변환은 **단일 책임 원칙(Single Responsibility Principle)**을 따라야 합니다. 결측치를 채우는 모듈, 데이터 타입을 맞추는 모듈, 비즈니스 지표를 집계하는 모듈로 잘게 쪼개야 합니다.
    
- 변환이 지원하는 비즈니스 규칙은 무엇인가?
    - 설명
        
        데이터 엔지니어는 단순히 데이터를 A에서 B로 옮기는 사람이 아니라, **비즈니스 규칙을 코드로 번역하는 사람**입니다.
