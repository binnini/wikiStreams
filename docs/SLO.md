# WikiStreams 서비스 수준 목표 (SLO)

> 작성일: 2026-03-06
> 최종 수정: 2026-03-07
> 상태: **1차 실측 baseline 반영** (관측 기간 진행 중 — 2026-04-06 이전 확정 예정)
> 선행 산출물: NFR.md, SLI.md
> 후행 산출물: SLA.md
> 측정 기간: 30일 롤링 윈도우
> baseline 수집 기간: 2026-03-06 KST ~ 2026-03-07 KST (약 39시간, 이벤트 3,565,064건)

---

## 1. SLO 개요

### 1.1 목적

실측 데이터 없이 NFR.md 요구사항에서 도출한 **초기 목표값**이다.
1~4주 관측 기간(Stage 4) 완료 후 baseline 데이터를 반영하여 수치를 검증·조정한다.

### 1.2 측정 기간

| 항목 | 값 |
|------|----|
| 롤링 윈도우 | 30일 |
| 최소 관측 기간 (SLO 확정 전) | 4주 |
| 리뷰 주기 | 월 1회 |

### 1.3 에러 버짓 계산 기준

```
에러 버짓(분) = 30일 × 24h × 60min × (1 - SLO 목표)
에러 버짓 소진률(%) = 실제 위반 시간 / 에러 버짓 × 100
```

소진률 75% 초과 시 Discord 경고 알림 발송.

### 1.4 외부 의존성 제외

아래 외부 서비스 장애로 인한 SLI 위반은 에러 버짓 소진에서 제외한다 (NFR.md §2 기준):
- Wikimedia SSE API 장애
- Wikidata REST API 장애
- Anthropic Claude API 장애
- Discord Webhook 장애
- Google News RSS 장애
- S3 Compatible Storage 장애

## 3. 성능 (Performance) SLO

### SLO-P1: 배치 처리 소요 시간

| 항목 | 값 |
|------|----|
| 연결 SLI | SLI-P1 |
| 연결 NFR | NFR-P1 |
| **목표** | ~~p95 ≤ 5초~~ → **p95 ≤ 2초 (캐시 히트 기준, 30일 롤링)** |
| 실측 baseline (Kafka) | p50=0.723s / **p95=1.208s** / p99=2.190s / max=32.594s (9,978배치) |
| 측정 도구 | Loki: `batch_processing_seconds` logfmt 필드 |
| 계측 현황 | 2026-03-06 계측 추가 완료 |

**목표 재조정 근거** (2026-03-07 1차 검증 리뷰):
- 실측 p95=1.208s → 기존 목표(5s) 대비 4배 이상 여유로 경고 의미 없음
- 2s로 하향: 실측 대비 +65% 허용 여유 유지하면서 실질적 성능 회귀 감지 가능
- Redpanda 전환 후에도 동일 목표 적용하여 회귀 여부 판단 기준으로 사용

---

### SLO-P3: ClickHouse 쿼리 응답 시간

| 항목 | 값 |
|------|----|
| 연결 SLI | SLI-P3 |
| 연결 NFR | NFR-P3 |
| **목표** | ~~p99 ≤ 1초~~ → **p99 ≤ 200ms (1시간 기준, 30일 롤링)** |
| 실측 baseline (Kafka + ClickHouse) | p50=2ms / p95=16ms / **p99=36ms** / max=259ms |
| 측정 쿼리 | `system.query_log` `quantile(0.99)(query_duration_ms)` |
| 측정 도구 | ClickHouse → Grafana SLO 대시보드 |

**에러 버짓**: p99가 200ms를 초과하는 쿼리 비율 ≤ 1%.

**목표 재조정 근거** (2026-03-07 1차 검증 리뷰):
- 실측 p99=36ms → 기존 목표(1,000ms)는 ClickHouse 장애에 가까운 수준에서만 위반
- 200ms로 하향: 실측 대비 5.5배 여유 유지, 의미 있는 성능 저하 감지 가능
- **아키텍처 경량화 참고**: 3단계(ClickHouse → DuckDB) 전환 시 이 SLO를 기준값으로 비교

### SLO-P5: Producer 지속 처리량

| 항목 | 값 |
|------|----|
| 연결 SLI | SLI-P5 |
| 연결 NFR | NFR-P5 |
| **목표** | ~~≥ 300 events/min~~ → **≥ 800 events/min (5분 이동 평균, 30일 롤링의 95% 시간대)** |
| 실측 baseline (Kafka) | p5=1,042 / p50=1,580 / p95=2,165 / max=2,407 events/min (1시간 버킷) |
| 측정 도구 | ClickHouse `wikimedia.events` count / interval |

**목표 재조정 근거** (2026-03-07 1차 검증 리뷰):
- 실측 최저(p5)=1,042로 기존 목표(300) 대비 3.5배 여유 → 사실상 무의미한 목표
- 800 events/min으로 상향: 실측 p5(1,042) 대비 -23% 허용 회귀 범위 설정
- Redpanda 전환 후에도 동일 기준으로 처리량 회귀 여부 판단

### SLO-P7: Wikidata 캐시 히트율

| 항목 | 값 |
|------|----|
| 연결 SLI | SLI-P7 |
| 연결 NFR | NFR-P7 |
| **목표** | **≥ 80% (10분 이동 평균, 30일 롤링의 90% 시간대)** |
| 실측 baseline | **93.14%** (3/6 KST 이후, Grafana 동일 방식) |

- 순수 Q-ID 단위 캐시 히트율(32.7%)과 다름 — Grafana 쿼리 기준을 SLI 정의로 확정

### SLO-D2: Wikidata 레이블 보강률

| 항목 | 값 |
|------|----|
| 연결 SLI | SLI-D2 |
| 연결 NFR | NFR-D2 |
| **목표** | ~~≥ 90%~~ → **≥ 80% (24시간 기준)** |
| 실측 baseline | **80.86%** (last 24h, Grafana 동일 방식) / **80.53%** (3/6 KST 이후) |
| 상태 | ⚠️ **목표 재조정 완료** — 경계값 수준, 지속 관찰 중 |
| 측정 쿼리 | `countIf(wikidata_label != '') / count()` WHERE `title LIKE 'Q%'` |
| 측정 도구 | ClickHouse → Grafana SLO 대시보드 |

**목표 재조정 근거** (2026-03-07 원인 분석):
- `title LIKE 'Q%'` 필터는 정확 — wikidata.org 이벤트의 Q-ID 형태 타이틀만 포함됨
- 빈 label 87,919건 중:
  - Q1억3천만 이상 신규 Q-ID: 33,354건 (37.9%) — 생성 직후 label 미부여 상태
  - 기존 Q-ID이나 en/ko label 없음: 54,565건 (62.1%) — Wikidata 자체에 레이블 없음
- 구조적 원인 (코드 개선 불가): Wikidata의 ~20% Q-ID 이벤트는 en/ko 레이블이 없는 엔티티
- 90% 목표는 비현실적, 실측 80.5~80.9% 기반으로 80%로 재조정

## 7. SLO 제외 항목

| SLI ID | 제외 이유 | 보강 필요 작업 |

## 8. 아키텍처 경량화 트레이드오프 SLO (t3.small 마이그레이션 로드맵)

### 8.1 목적 및 배경

| 항목 | 값 |
|------|----|
| 최종 목표 | AWS t3.small (2 GiB RAM) 마이그레이션 |
| 현재 메모리 | ~4,043 MiB (t3.small 197% — 초과) |
| 전략 | 4단계 경량화 로드맵으로 단계적 절감 |

단계별 완료 후 해당 SLI를 재측정하여 성능 회귀 여부를 수치로 확인한다.
각 단계의 SLO 달성 여부가 다음 단계 진행 조건이 된다.

---

### 8.2 메모리 절감 로드맵

| 단계 | 내용 | 절감 | 누적 메모리 | t3.small 비율 | 상태 |
|------|------|------|------------|--------------|------|
| 현재 | Kafka 기반 운영 스택 | — | ~4,043 MiB | 197% ❌ | 운영 중 |
| 1단계 | DLQ Consumer 제거 | -27 MiB | ~4,016 MiB | 196% ❌ | 대기 |
| **2단계** | **Kafka → Redpanda 전환** | **-802 MiB** | **~3,214 MiB** | **157% ❌** | **스테이징 검증 중** |
| 3단계 | ClickHouse → DuckDB 전환 | ~-1,882 MiB | ~1,332 MiB | 65% ✅ | 대기 |
| 4단계 | Loki + Alloy 제거 | ~-288 MiB | ~1,044 MiB | 51% ✅ | 대기 |

> Redpanda 전환(2단계) 단독으로는 t3.small 충족 불가. 3단계(ClickHouse → DuckDB)까지 완료해야 2 GiB 이내 진입.

---

### 8.3 2단계: Kafka → Redpanda 트레이드오프 SLO

브로커 교체로 영향받는 SLI에 한해 Kafka baseline과 Redpanda 목표를 비교한다.
브로커 비관련 SLI(A2, P7, D2, CAP1, CAP2)는 기존 목표를 그대로 적용한다.

#### 파이프라인 성능 SLI

| SLI | 항목 | Kafka 실측 (baseline) | Redpanda 목표 | 허용 회귀 | Redpanda 실측 (스테이징) |
|-----|------|-----------------------|--------------|----------|--------------------------|
| P1 | 배치 처리 p95 | 1.208s | **≤ 2.0s** | +65% | 0.69~0.75s ✅ |
| P2 | 배치 크기 p50 | 274개 | **≥ 200개** | -27% | 254~350개 ✅ |
| P5 | 처리량 p5 | 1,042 events/min | **≥ 800/min** | -23% | 관측 중 |
| D1 | 데이터 신선도 lag | 6s | **≤ 15s** | +150% | 13s ✅ |

#### 리소스 SLI (경량화 핵심 지표)

| 항목 | Kafka 실측 | Redpanda 목표 | Redpanda 실측 (스테이징) |
|------|-----------|--------------|--------------------------|
| 브로커 메모리 | 1,174 MiB | **≤ 500 MiB** | 372 MiB ✅ |
| 브로커 CPU p95 | ~1% | **≤ 5%** | ~0.4% ✅ |
| 전체 스택 메모리 | ~4,043 MiB | **≤ 3,500 MiB** | ~3,214 MiB ✅ (예상) |

#### 진행 조건 (Go/No-Go)

Redpanda 운영 전환은 아래 조건을 모두 충족해야 한다:
- [ ] SLI-P1 p95 ≤ 2.0s (7일 이상 관측)
- [ ] SLI-P5 p5 ≥ 800 events/min (7일 이상 관측)
- [ ] SLI-D1 lag ≤ 15s (7일 이상 관측)
- [ ] 브로커 메모리 ≤ 500 MiB (스테이징 현재 372 MiB ✅)
- [ ] SLO 관측 기간(4단계) 완료 후 전환

---

### 8.4 3단계 예고: ClickHouse → DuckDB 트레이드오프 SLO

3단계 완료 후 아래 기준으로 SLO를 재평가한다.
DuckDB는 in-process 라이브러리이므로 쿼리 응답 특성이 다를 수 있다.

| SLI | ClickHouse 기준 (현재 SLO) | DuckDB 검증 목표 |
|-----|--------------------------|-----------------|
| P3 쿼리 응답 p99 | 36ms (실측) / **≤ 200ms (SLO)** | ≤ 500ms (회귀 허용) |
| CAP1 메모리 사용률 | p95=26.22% (~3,661 MiB) | ≤ 30% of 2 GiB (≤ 600 MiB) |
| ClickHouse 스택 메모리 | ~2,169 MiB | DuckDB ≤ 300 MiB (목표) |
