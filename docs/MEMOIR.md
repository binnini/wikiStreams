## 왜 만들었나

"Wikimedia의 실시간 편집 스트림을 분석하면 전 세계적으로 어떤 주제가 주목받고 있는지 알 수 있지 않을까?" 라는 생각에서 시작했습니다.

단순한 아이디어를 넘어 기술적 질문으로 발전시켰습니다. "실제 생성되는 스트리밍 데이터를 최소 비용으로 운영하려면 어떤 문제를 해결해야 할까?" 이 질문에 답하고 싶었습니다.

구체적인 목표는 네 가지였습니다:

1. **카파 아키텍처**를 실제로 구현하고 운영하며 이해하기
2. **SLO/SLI 기반 설계 및 운영**을 경험하기 (정의만 아는 것과 운영해보는 것의 차이)
3. **제약 조건 속 의사결정** — 클라우드 비용이 현실이 될 때 어떤 트레이드오프를 선택하는지
4. **클라우드 환경 배포 후 운영 모니터링** — 운영 과정 중에 발생하는 문제가 무엇이고 어떻게 해결하는지

## 전체 아키텍처

```
Wikimedia SSE Stream
  → Producer (Python)        # 이벤트 수집 + Wikidata 레이블 보강 (SQLite 캐시)
  → Redpanda (Kafka 호환)     # 메시지 버스
  → QuestDB                  # 시계열 DB (TTL 5일)
  → Grafana                  # 대시보드 + SLO 모니터링 + 알림
  → Reporter (Python)        # 매일 09:00 KST Claude Haiku → Slack daily report
  → S3 Exporter              # 매일 01:00 UTC Parquet → S3 Datalake
```

모든 인프라는 Docker Compose로 코드로 관리(IaC)하며, AWS EC2 t4g.small 단일 호스트에서 운영합니다.

## 아키텍처 의사결정 4단계

이 프로젝트에서 가장 많이 배운 부분은 "무엇을 만들지"보다 "왜 그것을 선택해야 하는지"입니다.

초기 MVP는 엔터프라이즈 스택을 그대로 가져왔습니다:

```
Wikimedia SSE Stream
  → Producer (Python)
  → Kafka (JVM)           # 메시지 버스
  → DLQ Consumer          # Dead Letter Queue 처리
  → ClickHouse            # 시계열 OLAP DB
  → Loki + Alloy          # 로그 수집 스택
  → Grafana
```

각 컴포넌트는 개별적으로 합리적인 선택이었지만, 합산하면 전체 스택이 t4g.small 2GiB RAM의 197%(~4,043 MiB)를 사용했습니다. OOM이 반복 발생하며 컨테이너 구동 자체가 불가능한 상태였습니다. 분당 ~1,000건 규모의 단일 노드 홈랩에 엔터프라이즈 아키텍처를 그대로 적용한 것이 문제였습니다. 이후 4단계 경량화를 통해 스택 메모리를 77% 줄였습니다.

---

### 결정 1 — Kafka → Redpanda

**맥락**
초기 Kafka(JVM) + ClickHouse 스택이 t4g.small 2GiB RAM의 197%를 사용했습니다. OOM이 반복 발생하며 컨테이너의 구동 조차 불가능한 상태였습니다.

**고려한 대안**
- Kafka JVM 힙 축소 튜닝 → 처리량 저하, 근본 해결 아님
- Redis Pub/Sub → 구현이 단순하지만, 메시지 영속성이 없고 컨슈머 오프셋 추적이 불가능해 재연결 시 데이터 유실이 불가피
- Redpanda → Kafka API 완전 호환, Rust 구현체

Redis를 채택하지 않은 핵심 이유는 **내구성**이었습니다. SSE 재연결이나 Consumer 재시작 시 오프셋 기반으로 이어받기 어렵고, "전송했으나 처리 못한 메시지"를 보장할 방법이 없었습니다. Kafka 호환 브로커가 필요했습니다.

**선택 근거**
코드 변경 0줄로 전환 가능하다는 점이 결정적이었습니다. 실제로 `docker-compose.yml`에서 이미지 하나만 교체했습니다.

**Kafka vs Redpanda 측정 결과** (30분 구간 × 3, N=30,000 메시지)

| 지표 | Kafka | Redpanda | 변화 |
|---|---|---|---|
| RSS 평균 | 510 MiB | **316 MiB** | -38% |
| RSS 장기 패턴 | 선형 증가 (OOM 위험) | **수평 고정** | — |
| p99 latency (운영 부하) | 1.9ms | **1.7ms** | 유사 |
| p999 latency (운영 부하) | 6.3ms | **3.9ms** | -38% |
| p999 latency (2.8× 부하) | **13ms** (GC pause) | 3.8ms | -71% |
| 재시작 복구 시간 avg | 4.1s | **1.9s** | -54% |

**트레이드오프 수용**
분당 1,000건 규모의 단일 노드 환경에서 Kafka는 오버엔지니어링이었습니다. p99 레이턴시는 Redpanda와 거의 동일했지만, RSS가 38% 줄고 장기 메모리 패턴이 안정화됐습니다. 고부하 시 Kafka의 JVM GC pause가 p999에서 두드러지게 나타난 것과 달리 Redpanda는 부하와 무관하게 일관된 수치를 보였습니다. 메모리 절감만으로도 전환 근거는 충분했습니다.

---

### 결정 2 — ClickHouse → QuestDB

**맥락**
Redpanda 전환으로도 ClickHouse(1,467 MiB)가 여전히 RAM의 71%를 단독으로 차지했습니다. AWS t4g.small의 2GB RAM 환경에 맞추기 위해서는 메모리를 감소시킬 필요가 있었습니다.

**고려한 대안**
- ClickHouse 메모리 제한 (`max_server_memory_usage_ratio=0.5`) → 쿼리 성능 저하 감수
- TimescaleDB → PostgreSQL 기반으로 친숙하지만, 서버 자체 RSS가 400~600 MiB로 절감 효과가 불확실하고 시계열 압축 설정이 복잡
- DuckDB → 분석 쿼리 성능은 우수하나, Grafana 연동·TTL·시계열 SQL을 모두 직접 구현해야 함
- QuestDB → 시계열 특화, 낮은 메모리 풋프린트, Grafana 직접 연동 가능

**DuckDB를 채택하지 않은 이유**

이 프로젝트 규모(분당 ~1,000건, single-writer)에서 성능·동시성은 DuckDB로도 충분합니다. 채택하지 않은 이유는 운영 편의 부재였습니다. DuckDB는 PostgreSQL wire protocol이 없어 Grafana 연동에 별도 API 레이어가 필요하고, TTL도 직접 구현해야 하며, `SAMPLE BY` 같은 시계열 SQL 확장도 없습니다. QuestDB는 이 세 가지를 모두 내장으로 제공합니다.

**선택 근거와 실제 측정**

5일치 합성 데이터(~780만 건)를 주입하고 Reporter 실제 쿼리 3종으로 비교했습니다.

| 조건 | ClickHouse | QuestDB |
|---|---|---|
| warm p50 (Q3 GROUP BY) | 0.079s | **0.012s** (7× 빠름) |
| cold p95 (Q3 GROUP BY) | 0.117s | **3.572s** (30× 느림) |
| 메모리 RSS | 1,467 MiB | **381 MiB** (-74%) |

**트레이드오프**
QuestDB는 OS page cache를 직접 활용하는 mmap 기반이라, warm 상태에서는 ClickHouse보다 훨씬 빠르지만 cold read 시 page fault spike가 발생합니다. 즉, 메모리는 줄었지만 cold 쿼리 응답은 오히려 느려졌습니다.

**트레이드오프 수용 근거**
Reporter는 하루 1회 실행이고, Grafana의 장기간 조회 빈도가 낮아 대부분의 시간 page cache가 warm 상태를 유지합니다. 홈랩 수준에서 cold spike는 수용 가능한 트레이드오프라고 판단했습니다.

### 결정 3 — SLO 기반 운영 체계 수립

**맥락**
아키텍처 전환이 완료된 시점에서 "이 시스템이 잘 동작하고 있는가"를 판단하는 기준이 없었기에 정량적으로 평가하고 싶었습니다.

**구현한 것**

11개 SLO 항목을 정의하고 QuestDB + Grafana 기반 대시보드를 구축했습니다.

| 영역 | SLO | 목표 | 현재 |
|---|---|---|---|
| 가용성 | A2 쿼리 성공률 | ≥ 99% | 수집 중 🔄 |
| 가용성 | A3 Reporter 발송 | 월 ≥ 27회 | 정상 ✅ |
| 성능 | P1 배치 처리 p95 | ≤ 2s | **0.72s ✅** |
| 성능 | P3 쿼리 응답 p99 | ≤ 200ms | **4~30ms ✅** |
| 성능 | P5 처리량 | ≥ 800/min | **~1,000/min ✅** |
| 성능 | P7 캐시 히트율 | ≥ 80% | **94% ✅** |
| 신뢰성 | R1 DLQ 비율 | ≤ 1% | **~0% ✅** |
| 데이터 | D1 데이터 lag | ≤ 30s | **~9s ✅** |
| 데이터 | D2 레이블 보강률 | ≥ 80% | **79~88% ⚠️** |
| 용량 | CAP1 QuestDB 메모리 | ≤ 880 MiB | **285~500 MiB ✅** |
| 용량 | CAP2 Producer CPU | ≤ 70% | **1~7% ✅** |


SLO 측정 결과 R1(DLQ 비율)이 near-zero(인프라 장애 시에만 간헐 발생)로 나타났습니다. 전용 DLQ Consumer 컨테이너를 유지하는 것이 실질적인 필요 없이 메모리만 소비하는 구조라 판단하여 제거했습니다. DLQ 이벤트는 토픽에 기록되므로 필요 시 수동 재처리가 가능합니다.

### 결정 4 — Loki/Alloy 제거

**맥락**
로그 수집 스택(Loki + Alloy)이 174 MiB를 사용하면서도 실질적으로 활용하지 않았습니다. SLO 지표를 Loki에 저장하던 구조를 QuestDB로 이관하면 로그 스택 전체를 제거할 수 있었습니다.

**결과** (EC2 실측, benchmark/loki-removal/ 기준)

| 항목 | Before | After |
|---|---|---|
| 컨테이너 수 | 10개 | 8개 |
| CPU 합계 | 30.6% | **13.3%** (-17%) |
| Swap | 700 MiB | **205 MiB** (-71%) |
| 이미지 디스크 | — | 778MB 추가 절감 |

CPU가 17% 감소한 주원인은 Loki/Alloy 자체(1%)가 아니라, **이들이 유발하던 로그 스트림 처리 부하**(producer/redpanda/questdb)가 함께 사라진 연쇄 효과였습니다. Swap 역시 700 MiB에서 205 MiB로 대폭 줄어들며 OOM을 방지할 수 있었습니다.

## 기술적 도전과 해결

### 1. 실시간 보강 파이프라인에서 외부 API 병목 해결

**맥락**
데이터 원천 분당 ~1,000건의 이벤트는 Wikidata Q-ID(`Q12345` 형식)만 포함되고 실제 제목이 표시되지 않아 어떤 페이지에 대한 기록인지 알 수 없었습니다. 

Wikipedia 레이블을 붙여 의미 있는 데이터로 보강해야 했습니다. 문제는 Wikidata API 호출이 이벤트마다 발생하면 배치 처리 시간이 외부 API 응답 시간에 종속된다는 점이었습니다.

**설계 결정**

세 가지 구조를 검토했습니다:
- 이벤트마다 개별 API 호출 → 레이턴시 누적, 처리량 급감
- 비동기 별도 보강 서비스 → 파이프라인 복잡도 증가, 데이터 일관성 관리 필요
- **배치 내 캐시 우선 조회 + 미스 시 일괄 API 요청** → 채택

**구현 핵심**
```
배치 수신 (500건) → Q-ID 추출 → SQLite 캐시 일괄 조회
  → 캐시 히트: 즉시 레이블 적용
  → 캐시 미스: Wikidata API에 최대 50개씩 묶어 일괄 요청 후 캐시 저장
```

캐시 TTL은 엔티티 상태에 따라 3가지로 분리했습니다:
- 정상 엔티티: **30일** (안정적인 레이블, 긴 TTL)
- missing 엔티티 (Q-ID 자체가 없는 경우): **3시간** (삭제됐다가 복구될 수 있음)
- 빈 레이블 엔티티 (레이블이 없는 Q-ID): **3시간** (추후 레이블 추가 가능)


실제로 초기에 missing/빈 레이블을 동일하게 30일로 설정하여 나중에 추가된 Q-ID를 오랫동안 빈 값으로 남겨 캐시 히트율이 60% 아래로 떨어졌었습니다. 이에 각 케이스 별로 적절한 TTL 값을 설정하여 히트율을 80% 이상으로 유지하고자 했습니다.

**결과**
- 배치 처리 p95: 0.72s (SLO P1 목표 ≤2s 달성)
- 캐시 히트율: 평균 94% (SLO P7 목표 ≥80% 달성)
- Wikidata API 호출 수: 배치당 평균 10~20회 (1,000건 대비 ~2%)

### 2. QuestDB 메모리 leak 의심 → mmap 동작 이해

**증상**: QuestDB RSS가 선형으로 계속 증가 (전환 직후 387MiB → 26시간 후 1,273MiB, +34MiB/h)

**오판**: 처음엔 메모리 누수를 의심했습니다.

**실제 원인**: ClickHouse의 내장 LRU 캐시와 달리 QuestDB는 컬럼 파일을 mmap으로 매핑하며, OS page cache를 직접 활용합니다. 데이터가 쌓일수록 파일 크기가 증가하고 mmap'd pages가 RSS에 반영됩니다. OS는 메모리 압력이 생기면 cold pages를 자동 evict합니다. 그러나 개발 환경(mac mini, RAM 24GB)에서 압력이 생기지 않았기에 cache를 정리하지 않았기 때문이었습니다.

**대응**: 누수가 아니므로 코드 수정이 아닌 운영 대응으로 해결했습니다.
- `mem_limit: 1100m` — cgroup 제약으로 OS eviction 유도 (t4g.small 규모)
- TTL 5d — 데이터 볼륨 상한으로 mmap working set 간접 제한

**배운 점**
이 과정에서 ClickHouse의 명시적 LRU 캐시 vs QuestDB의 OS page cache 위임 방식의 트레이드오프를 깊이 이해하게 됐습니다.

### 3. S3 Exporter 메모리 스파이크 — 시간별 청크 처리로 개선

**맥락**
S3 Exporter는 매일 01:00 UTC에 QuestDB에서 하루치 데이터를 내려받아 Parquet으로 변환 후 S3에 업로드하는 구조였습니다. t4ㅎ.small(1.85GB) 환경에서 실측 시 메모리 스파이크가 과도하게 발생했습니다.

**원인 분석**
```
_fetch_csv()    : 하루치 CSV 전체를 resp.read()로 한 번에 메모리 로드 (~800MB/day)
_csv_to_parquet(): CSV bytes + PyArrow Table + Parquet bytes 3가지 동시 점유
```
2.5시간치 실측으로 추산하면 하루치 스파이크가 ~960MB. 기존 프로세스들(QuestDB 376MB + Redpanda 341MB 등)이 이미 ~1.1GB를 점유한 상태에서 OOM이 불가피했습니다.

**검토한 대안**
- Producer에서 직접 S3 write → Parquet은 컬럼형 포맷으로 스트리밍 write에 부적합, 소파일 문제
- Redpanda S3 Consumer 신규 추가 → 아키텍처 복잡도 증가, 오버엔지니어링
- Swap 1GB 추가 → page fault로 메인 프로세스 성능 저하, 근본 해결 아님

**결정 및 구현**
`s3_exporter/main.py` 내에서 1시간 단위 청크로 QuestDB를 조회하고, `ParquetWriter`로 순차 write한 뒤 S3 업로드는 1회만 수행하도록 변경했습니다.

```
기존: 하루치 전체 조회 → CSV → Parquet 변환 → S3 1회
변경: 1시간씩 × 24회 조회 → ParquetWriter 순차 write → del table (즉시 해제) → S3 1회
```

**결과**

| 항목 | Before | After |
|---|---|---|
| 2.5시간치 실측 스파이크 | ~100MB | **~50MB** (-50%) |
| 하루치 추정 스파이크 | ~960MB | **~480MB** |
| S3 업로드 횟수 | 1회 | **1회 유지** |

서버 여유 메모리 ~750MB 내에서 OOM 없이 통과 가능한 수준이 됐습니다.


---

### 4. Redpanda Disk Full → 9시간 파이프라인 중단

**증상**: QuestDB로의 이벤트 데이터 적재 중단. Producer가 Redpanda 연결 끊김 에러 발생.

**원인**: Redpanda 토픽에 Retention 정책을 설정하지 않아 메시지가 무한 누적되었습니다. 파이프라인 가동 이후 토픽이 ~1.25GB까지 성장해 디스크 100%를 채웠고, Redpanda가 쓰기를 거부하면서 파이프라인 전체가 연쇄적으로 멈췄습니다.

**조치**:
- Redpanda retention을 약 24시간 수준으로 설정하여 주기적으로 디스크를 확보하도록 하였습니다.
- 비활성 볼륨(Loki, ClickHouse 잔재) 정리로 디스크 6GB 확보

**배운 점**: 단일 노드에서 디스크 풀은 즉각적인 전체 장애로 이어집니다. Retention 정책은 브로커 설계 시 가장 먼저 설정해야 하는 항목이었습니다. 또한 9시간이 지나서야 장애를 발견했다는 점에서, 모니터링 알림 및 자동 복구 오케스트레이션의 필요성을 깨달았습니다.

## 운영 경험에서 배운 것

### 단일 노드에서 99% 가용성 달성의 어려움

Disk Full 장애 하나로 9시간이 다운됐습니다. 단일 노드에서 99% 이상의 가용성을 달성하고자 했으나, 그러려면 **사람이 개입하지 않아도 자동 복구되는 구조가 전제**돼야 합니다. 또한 **헬스체크 기반 자동 재시작, 복제본, 처리량 기반 알림이 없으면 구조적으로 한계가 있다는 것을 운영을 통해 직접 확인했습니다.**

### 개발·운영 환경 분리의 필요성

EC2에서 Claude Code CLI를 실행하다 OOM이 발생해 Docker 컨테이너가 연쇄적으로 불안정해진 적이 있었습니다. 사소한 작업 하나가 프로덕션 파이프라인에 영향을 미쳤습니다. **개발, 테스트, 운영 환경을 완벽히 격리하지 않으면 예상치 못한 방향에서 장애가 발생한다는 것을 배웠습니다.**

### 비용 절감과 성능 사이의 트레이드오프

아키텍처 경량화 4단계를 거치면서, **비용을 줄이는 것이 단순히 더 작은 인스턴스로 이전하는 문제가 아님을 배웠습니다.** 각 구성 요소가 메모리를 얼마나 사용하는지, 어떤 상황에서 급증하는지, 어떤 설정으로 상한을 걸 수 있는지를 이해해야 했습니다. QuestDB의 mmap 동작, Redpanda의 메모리 안정성, Loki의 연쇄 부하처럼 직접 측정하기 전까지는 알 수 없었던 것들이 있었습니다.

---

### 숫자 없이는 아무것도 모른다

Kafka + ClickHouse로 MVP를 구축한 뒤, 이 시스템이 잘 동작하는지 어떤 부분을 개선해야 하는지 파악하기가 어려웠습니다. Kafka vs Redpanda, ClickHouse vs QuestDB, Loki/Alloy 필요성 같은 프레임워크 간 비교도 실제로 측정하기 전까지는 그저 문장일 뿐 와닿지 않았습니다. p999 GC pause, cold read spike, mmap RSS 증가 — 이런 것들은 직접 부하를 걸고 수치를 뽑은 뒤에야 비로소 실감할 수 있었습니다.

### SLO의 목적은 알림이 아니라 판단 기준

SLO는 **프로젝트의 완성도 및 방향성을 판단하는 기준**이라는 걸 알게 됐습니다. 아키텍처 전환 및 이상 감지 등 프로젝트의 현황과 방향성을 재검토해야 하는 상황이 생겼을 때, SLO가 있었기에 그 논의를 시작할 수 있었습니다.

### 문서는 코드만큼 중요하다

41개 이상의 DEV_LOG 항목, 아키텍처 경량화 보고서, 벤치마크 원본 데이터를 모두 저장소에 함께 관리했습니다. 몇 주 후에 "왜 이렇게 했지?"라는 질문에 스스로 답할 수 있었습니다.

- **CI 테스트 단계 분리**: 현재 PR마다 276개 전체 테스트를 실행합니다. Smoke test(PR) / Full E2E(merge) 분리가 필요합니다.
